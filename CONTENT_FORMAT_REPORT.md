# 보라-부산(bora-busan) 콘텐츠 포맷 및 데이터 구조 분석 리포트

조사해주신 '아이돌 성지순례 표준 포맷'은 팬들의 실질적인 니즈(이동 시간, 인증샷 위치, 굿즈 구매 등)를 정확히 찌르는 매우 훌륭한 구조입니다. 

'보라-부산(BTS 등 아이돌 부산 성지순례 특화 예상)' 앱의 핵심 경쟁력은 **'얼마나 예쁘고 보기 좋게 정보를 제공하는가'**와 **'얼마나 쉽게 따라 할 수 있는가'**에 있습니다. 조사하신 내용을 바탕으로 우리 앱에 어떻게 적용하면 좋을지 기획 및 기술적 관점에서 면밀히 분석했습니다.

---

## 1. 콘텐츠 타입의 이원화 (투트랙 전략)

조사하신 내용처럼 콘텐츠는 크게 두 가지 타입으로 분리하여 작성 포맷을 다르게 가져가야 합니다.

### Type A: 단일 장소 스팟형 (Point of Interest)
특정 카페, 촬영지, 사옥 등 한 장소에 대한 깊이 있는 정보 제공.
*   **작성 포맷**: 블로그 본문 상단에 **[요약 정보 카드 UI]**를 고정 배치하고, 하단에 상세 사진과 텍스트를 풀어내는 방식.
*   **필수 포함 데이터**: 
    *   카테고리 (카페/식당/관광지 등)
    *   영업시간, 휴무일 (매우 중요)
    *   **인증샷 꿀팁** (어느 자리, 어떤 각도)
    *   지도 좌표 (카카오/네이버 지도 링크)

### Type B: 당일/다일 코스형 (Timeline Course)
스팟들을 엮어서 하루 또는 1박 2일 일정으로 제공.
*   **작성 포맷**: 본문 상단에 **[코스 개요 및 총 예상 비용/시간]**을 보여주고, 하단에는 **[타임라인 UI]**를 활용하여 동선을 시각화.
*   **필수 포함 데이터**:
    *   스팟 간 이동 수단 및 소요 시간 (예: 버스 15분, 도보 5분)
    *   시간대별 일정표
    *   총 예상 경비

## 2. Supabase DB 스키마 및 데이터 구조 확장 제안

현재 `posts` 테이블은 `title`, `description`, `category`, `image_url` 등 기본적인 형태만 갖추고 있습니다. 조사하신 JSON 구조를 반영하려면 DB 스키마를 고도화해야 합니다.

### 대안 1: JSONB 컬럼 활용 (빠른 구현)
기존 `posts` 테이블에 `metadata`라는 `JSONB` 타입의 컬럼을 하나 추가하여 모든 상세 정보를 넣습니다.
```json
// posts.metadata 컬럼 데이터 예시
{
  "type": "place", // or "course"
  "address": "부산광역시 금정구...",
  "coordinates": { "lat": 35.23, "lng": 129.08 },
  "tips": ["창가 자리가 채광이 좋아요", "주말엔 오픈런 필수"],
  "tags": ["지민", "맛집", "오션뷰"]
}
```

### 대안 2: 테이블 정규화 (장기적 확장성)
앱이 커지면 장소 검색, 지도 핀 표시 등이 필요해집니다.
*   `posts` 테이블: 블로그 글 자체 (제목, 본문 마크다운, 작성일 등)
*   `places` 테이블: 장소 마스터 데이터 (이름, 주소, 위도/경도, 영업시간 등)
*   `post_places` 테이블 (M:N): 코스형 글을 위해 하나의 글에 여러 장소를 연결.

**▶ 기획자 추천**: 초기 런칭 시점에는 **대안 1 (JSONB 활용)**로 빠르게 구현하면서 유연하게 포맷을 변경해 보고, 데이터가 쌓이고 지도 뷰(Map View) 기능을 추가할 때 대안 2로 마이그레이션 하는 것을 추천합니다.

## 3. 프론트엔드 UI/UX 구현 방안 (React/Tailwind)

조사하신 포맷을 텍스트(줄글)로만 보여주면 가독성이 떨어집니다. 웹앱의 장점을 살려 **컴포넌트화**해야 합니다.

1.  **InfoCard 컴포넌트**: 장소 글 최상단에 배치되는 요약 카드. 아이콘(시간, 주소, 지하철)을 활용해 한눈에 들어오게 디자인.
2.  **Timeline 컴포넌트**: 코스 글에서 세로 선을 기준으로 시간대별 스팟이 이어지는 UI. (왼쪽엔 시간, 오른쪽엔 스팟 정보)
3.  **Badge/Tag 컴포넌트**: 조사하신 "초보 추천", "혼잡도" 등을 태그 형태로 시각화. (현재 `badge_type`이 구현되어 있으므로 이를 확장)

## 4. 관리자(Admin) 작성 환경 가이드

이러한 구조화된 포맷을 관리자가 일일이 마크다운 표기로 작성하면 오타가 나거나 UI가 깨질 위험이 큽니다.

*   **해결책**: 관리자 페이지의 글쓰기(`src/app/admin/(protected)/write`) 폼을 탭(Tab)으로 나누어 구성합니다.
    *   **Tab 1 (기본 정보)**: 제목, 썸네일, 카테고리
    *   **Tab 2 (메타 데이터)**: 장소 주소, 지도 좌표, 태그, 팁 등을 입력하는 입력 폼(Input Fields) 제공. (여기서 입력된 값이 JSONB로 저장됨)
    *   **Tab 3 (본문 작성)**: 감성적인 사진과 에세이를 적는 마크다운 에디터.

## 📝 결론 및 액션 아이템

조사해주신 포맷은 **"팬들의 실제 행동 패턴을 완벽히 이해한 가이드"**입니다. 이를 시스템에 녹여내기 위해 다음 작업을 우선순위로 진행할 것을 제안합니다.

1.  **DB 업데이트**: Supabase `posts` 테이블에 `metadata` (JSONB) 컬럼 추가.
2.  **컴포넌트 개발**: 상세 페이지용 `<PlaceInfoCard />` 및 `<CourseTimeline />` UI 컴포넌트 퍼블리싱.
3.  **관리자 폼 수정**: 에디터에 주소, 팁, 좌표 등을 입력할 수 있는 필드 추가.
